# Review Checks

在审查 easy-query 业务代码，且 MCP 证据不足时，使用本文件作为本地降级检查清单。

## 核心检查项

- 当前结论属于源码事实，还是本地参考建议
- 查询是否优先走主键、唯一键或显式条件
- `whereObject` 是否被误用到复杂跨表业务逻辑
- 分页是否具备稳定排序
- 动态排序是否做字段白名单
- 更新/删除条件是否完整
- 0 行更新是否被当成异常业务信号处理
- 事务边界是否收敛在 Service 层
- 是否存在先查后改且没有并发保护
- 是否把 easy-query DSL 泄漏到不该泄漏的层
- `any()` / `count()` 是否被错误替换成全量 `toList()`
- `whereByIds`、`in(...)` 这类集合条件是否先做了空集合保护
- 状态流转更新是否把旧状态一起带进条件
- 幂等写入是否真正依赖唯一键或冲突处理，而不是纯应用层判断
- 事务内是否夹带远程调用、通知、文件 IO 或其他长耗时步骤
- `sum/max/min/group by` 是否被错误放到了内存层而不是 SQL 层
- `having`、`distinct`、`exists` 子查询这类语义是否仍被错误拆成多次内存处理
- 联表分页是否仍然保留了稳定排序和可解释的投影边界
- join 条件、子查询条件、分组字段是否和 DTO 字段一一对应
- 是否存在“主表列表 + 循环 count/sum 子查询”的 N+1 聚合写法
- `union`、`conflict`、`upsert`、逻辑删除恢复这类强版本/方言能力，是否被误当成已证实 API
- 更新表达式是否能替代先查后改，避免无谓的并发窗口
- 查询对象、筛选对象是否只承载主表可控条件，而不是塞进复杂跨表业务逻辑
- 对象排序是否做了白名单映射，而不是直接透传前端字段
- `PARTITION BY`、递归树查询、大数据流式查询、结构化属性填充这类能力，是否被误写成已验证 API
- `map` 返回是否明确字段名、字段类型与上层消费边界
